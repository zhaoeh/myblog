# 1. 数据库事务使用中应该遵循的原则
事务越短越好。      
这不是一句空话，具体可以描述为：事务只包含必须原子性的一组数据库写操作，其他一切（查询、校验、远程调用、缓存操作等）原则上都应该放在事务外。         

# 2. 为什么“事务要短”不是一句空话？      
长事务（大事务）的代价其实非常具体：      
（1）持有数据库锁时间变长：更容易死锁。      
（2）占用连接时间长：连接池更容易被打满。      
（3）回滚成本高：undo log膨胀。      
（4）并发下降：TPS直接被拖垮。      
因此，事务要“短”，不是优化，是系统能不能扛住流量的生死线。      


# 3. 正确的事务拆分原则
可以使用一个非常实用的分层机制：      

## 3.1 事务外（准备阶段）。    
这些操作绝对不要进事务：       
（1）参数校验。    
（2）业务规则判断。    
（3）查询数据（只读）。      
（4）远程调用接口（HTTP/RPC等）。      
（5）查询redis/mogo等。      
（6）计算逻辑等。      
原因很简单：这些操作都不需要回滚，放进事务内纯属拖慢。      

## 3.2 事务内（核心阶段）
只做三件事情：      
（1）必要的查询（for update或者强一一致依赖）：有的场景需要在事务内再做一次查询，保证事务隔离级别，确保事务的不可重复读。    
（2）数据写入（insert/update/delete）。      
（3）保证数据一致性的判断。    

这个阶段要做到：      
（1）无远程调用。    
（2）无复杂逻辑。    
（3）无长时间计算。    

## 3.3 事务后（收尾阶段）
事务提交后再做：      
（1）更新redis。     
（2）发送MQ（推荐）。    
（3）调用其他服务。    
（4）写日志。     
（5）触发异步任务。    

## 3.4 一个典型的错误流程 vs 正确流程
***错误写法（非常常见）***   
```java
@Transactional
public void createOrder() {

    // 1. 查库存（OK）
    checkStock();

    // 2. 调远程库存服务 ❌
    remoteService.lockStock();

    // 3. 写订单
    orderMapper.insert();

    // 4. 写 Redis ❌
    redisTemplate.set(...);

    // 5. 调支付服务 ❌
    payService.call();
}
```
存在的问题：      
（1）远程调用慢：事务占有时间被拉长。    
（2）Redis失败：事务白回滚。    
（3）外部系统不可控：拖死数据库。    

***推荐写法（工程实践）***      
```java
public void createOrder() {

    // 1. 前置校验（事务外）
    checkStock();

    // 2. 远程调用（事务外）
    remoteService.lockStock();

    // 3. 核心事务
    doCreateOrder();

    // 4. 事务后处理
    afterCommit();
}

@Transactional
public void doCreateOrder() {
    orderMapper.insert();
    stockMapper.update();
}
```

# 四、Redis / MQ / 外部系统 怎么处理？

这是重点。

## 1️⃣ Redis

👉 原则：**Redis 不是事务的一部分**

错误思路：

* 想让 Redis 和 DB 强一致（基本做不到）

正确做法：

### 方案 A：先写 DB，后删缓存（经典）

```text
DB成功 → 删除缓存
```

### 方案 B：延迟双删（高并发）

```text
删缓存 → 写DB → 延迟再删缓存
```

---

## 2️⃣ MQ（推荐解耦神器）

👉 标准做法：**事务提交后再发 MQ**

否则会出现：

* DB回滚了，MQ已经发出 ❌

### 正确方式：

#### 方式1：事务后回调

```java
TransactionSynchronizationManager.registerSynchronization(...)
```

#### 方式2（更好）：本地消息表（最终一致性）

* DB事务中：

  * 写业务数据
  * 写消息表
* 后台任务投递 MQ

👉 这是电商、支付系统标准解法

---

## 3️⃣ 远程调用（RPC / HTTP）

👉 原则：

* **绝不能放在事务里**
* 必须考虑失败补偿

常见模式：

### Saga / 补偿模式

```text
A服务成功 → 调B服务
如果B失败 → 回滚A（补偿）
```

# 二、什么时候你的方案是“合理的”？

只有在下面这种情况下才成立：

## ✅ 场景1：三方操作是“预操作（可撤销）”

比如：

* 远程锁库存（但可以解锁）
* 预占额度（但可以释放）
* 创建草稿订单（但可以取消）

流程：

```text
1. 调三方：预占资源 ✅
2. 本地事务提交成功 ✅
3. 如果失败 → 调三方释放资源（补偿）
```

👉 这个模式其实叫：

> **Try-Confirm-Cancel（TCC）**

---

## ✅ 场景2：三方操作是“幂等且无副作用”

比如：

* 调用一个“查询类接口”
* 或者“重复调用不会改变状态”的接口

那就无所谓顺序

---

## ❌ 但大多数情况是不满足的

比如：

* 扣款
* 发券
* 发货
* 创建订单（外部系统）

👉 这些一旦成功，你**没法优雅回滚**

```text
开事务 → 写DB → 最后才发现库存不够 ❌
```

---

## 🔴 类型2：**有副作用（Do类）**

比如：

* 扣库存
* 扣钱
* 发券
* 调用三方创建资源

特点：

* 一旦成功就改变状态
* 不一定能回滚
* 可能失败

👉 这些操作：

> ❗ **才是你说的那类：必须在事务之后**

---

# 三、所以正确模型不是“是否脱离JVM”，而是：

> 🔥 **是否产生“不可逆副作用”**

---

# 四、帮你把认知升级成最终版本（建议你记这个）

```text
1. Check（无副作用） → 必须前置
2. 本地事务（DB） → 必须作为锚点优先提交
3. Do（有副作用） → 必须在事务后执行
4. 所有Do操作必须具备：幂等 + 重试 + 补偿能力
```

---

# 五、为什么“只看是否脱离JVM”是危险的？

我举个反例你就懂了：

---

## ❌ 如果你坚持“所有三方都后置”

```text
1. 本地事务提交（创建订单）
2. 再去查库存（远程）
```

👉 结果：

* 订单创建成功
* 但库存其实不够

👉 数据直接错误 ❌

---

## ✅ 正确做法

```text
1. 查库存（Check）
2. 本地事务创建订单
3. 扣库存（Do）
```

---

# 六、再帮你把你那句话“修正一下”

你原话：

> “所有脱离当前JVM的行为都应该在本地事务之后做”

👉 改成：

> ✅ **所有“不可控副作用”的操作，都必须在本地事务之后做**
> ❗ **所有“校验类”的操作，必须在事务之前完成**

---

# 七、再补一个你现在还没完全踩到的坑（很重要）

即使你把 Do 放在事务后：

```text
DB成功 ✅
三方失败 ❌
```

👉 你还是要解决：

* 怎么重试？
* 怎么避免重复执行？
* 怎么保证幂等？

否则系统还是不稳定。

---

# 八、所以最终完整闭环应该是

```text
【事务外】
- Check（校验）

【事务内】
- 写DB（锚点）

【事务后】
- Do（三方调用）
  - 成功 → 更新状态
  - 失败 → 标记 + 重试

【后台机制】
- 定时任务 / MQ 重试
- 幂等控制
```

> ❗ 不是“是否脱离 JVM 决定顺序”
> ✅ 而是“是否产生不可逆副作用决定顺序”
